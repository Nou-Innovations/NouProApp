## 1. 🔴 The Shipping Gap (your biggest problem)

**This is the most important finding in the whole audit.** You are fixing bugs that never reach your users.

Sentry shows **4 unresolved crashes** hitting real users. I checked each crash site against your current code:

| # | Crash | Where | Fixed in your code? | Still crashing users? |
|---|---|---|---|---|
| MOBILE-2 | `ReferenceError: Property 'ScrollView' doesn't exist` | `NewChatModalList.tsx` | ✅ Yes (uses imported `AppBottomSheetScrollView`) | ❌ Yes |
| MOBILE-1 | `ReferenceError: Property 'center' doesn't exist` | `AppBottomSheet.tsx` | ✅ Yes (no bare `center` remains) | ❌ Yes |
| MOBILE-3 | `ReferenceError: Property 'useNotifications' doesn't exist` | `HomeScreen.tsx` (personal) | ✅ Yes (no longer references it) | ❌ Yes |
| MOBILE-4 | `NSInternalInconsistencyException: touch registries` | native iOS (no JS culprit) | ⚠️ Native gesture/nav race — 1 event, 1 user | Occasional |

**The pattern:** 3 of 4 crashes are the *same class of bug* — a symbol used but never imported. Hermes only throws these **at runtime**, so dev mode and `tsc` (which has 156 errors and isn't gating) don't catch them. All three are **committed and clean** in your repo already. They keep crashing because **the production OTA channel is running an older JS bundle.**

### What to do
1. **Publish the current bundle to production** (`eas update --branch production`, run on your Mac — the sandbox can't reach Expo's servers). This kills MOBILE-1/2/3 immediately for anyone who already has the app.
2. **Add a runtime safety net** so this class of bug can't ship silently again:
   - Turn on a **`tsc --noEmit` gate in CI** once the 156-error backlog is burned down (a bare-identifier reference is a `tsc` error — CI would have caught all three).
   - Until then, add an **`eslint` rule `no-undef`** (catches undefined identifiers) and make it error-level.
3. **MOBILE-4** (native touch registry) is a known React Native issue when touches land during a navigation/modal transition. Low frequency; watch it after the OTA update. If it climbs, pin `react-native-screens`/`react-native-gesture-handler` versions and check for modals dismissed mid-gesture.

## Security Audit

### 🔴 SEC-1 — Payment webhook is forgeable
- **Where:** `backend/server.js:14253` (`POST /api/webhooks/peach`)
- **Verified state:** The handler now uses `express.raw()` (the prerequisite for signature checks) **but performs no signature/HMAC verification** and **never re-fetches the authoritative result from Peach**. It reads the success code straight from the POST body (`server.js:14276`) and, on a success code, calls `processSuccessfulPayment` → activates the subscription (`:14329`) or marks the invoice paid (`:14340`).
- **New partial mitigation vs. June:** it now requires a **pre-existing `Payment` row** matching `peachCheckoutId` (`:14263`) and skips already-processed payments. So an attacker can no longer invent an arbitrary checkout ID.
- **Still exploitable:** an attacker starts a *real* checkout (creating a legitimate pending `Payment` + `checkoutId`), abandons it **without paying**, then POSTs a forged body `{ payload: { checkoutId, result: { code: "000.100.110" } } }`. The code matches the pending payment, sees a success code, and **grants them the plan / marks the invoice paid.** Self-upgrade to ENTERPRISE for free.
- **Fix:** (1) Verify the Peach webhook signature against `PEACH_WEBHOOK_SECRET`. (2) **Do not trust the body** — call the already-implemented `getCheckoutResult(checkoutId)` in `peachPayments.js` and use *that* authoritative status to decide. Both, ideally.

### 🔴 REV-1 — Recurring billing never charges the card
- **Where:** `backend/src/services/peachPayments.js:116` (`chargeStoredCard`) — **zero callers** (only its definition and export).
- **Verified state:** The two `setInterval` schedulers in `server.js:14472,14480` are (a) recurring **deliveries/transfers** and (b) daily **stuck-order alerts** — neither charges anything. The card token (`peachCardRegistrationId`) is stored and "renews on X" notifications are sent, but no job ever bills.
- **Impact:** First payment succeeds; subscription lapses silently at `currentPeriodEnd`; recurring revenue is zero.
- **Fix:** Add a renewal job that finds businesses past `currentPeriodEnd` with a stored card and calls `chargeStoredCard`, exposed as an automation endpoint protected by `AUTOMATION_API_KEY` (mirror the order-automation pattern), triggered by an external cron (Render Cron / cron-job.org). This is revenue, not a user-facing crash — sequence it right after the webhook fix.

### 🟠 SEC-3 — Input validation is only partial
- **Verified state:** `zod` is imported (`server.js:17`) but there are only **5 `validateBody()` call sites** across ~200 routes. The vast majority of endpoints write request-body fields into Prisma with no schema validation.
- **Impact:** Malformed/oversized/unexpected fields reach the DB layer; some become 500s, some silently persist junk. Not a data-leak per se, but a reliability and abuse surface (e.g. unbounded string fields, missing pagination caps → resource exhaustion).
- **Fix:** Add `validateBody(schema, …)` to the high-value write routes first: auth/register, order/invoice/product create+update, payments, profile update. You already have the helper and the pattern — it's incremental.

### 🟡 SEC-4 — Notes / lower risk
- **CORS/helmet:** helmet is on; confirm `CORS_ORIGIN` in prod is your real origin, not `*`.
- **Frontend logging:** 190 `console.*` in `src/` are stripped from prod by the babel plugin — but double-check nothing logs tokens/PII in a path that runs before the plugin applies.
- **Secrets:** clean — `.env` gitignored, no committed secrets. The tracked `ios/.xcode.env` files are standard RN build config (safe, not secrets).

### Security posture summary
The June audit's security P0s are **closed** except the two payment issues above. Auth, hashing, 2FA, rate limiting, tenancy scoping, and security headers are all in good shape. **The remaining security work is essentially just the two payment items + broadening input validation.**

> ⚠️ **Confidence note:** A route-by-route IDOR/tenancy re-verification (every `findUnique`/`update` scoped by company) was *not* exhaustively repeated this session — the deep agent pass was interrupted by a backend outage. ## Release Readiness — Path to App Store

### 🔴 REL-1 — iOS permission usage strings missing (App Store blocker)
- **Verified:** No `NSCameraUsageDescription` / `NSPhotoLibraryUsageDescription` strings in `app.json` or `app.config.ts`. Your app uses the camera and image picker.
- **Impact:** iOS **crashes** when the permission is requested without a description, and **Apple rejects** apps missing them.
- **Fix:** Add `ios.infoPlist` usage strings (camera, photo library, and photo-library-add), plus a notifications rationale. If you rely on the `expo-image-picker`/`expo-camera` config plugins, set explicit `photosPermission`/`cameraPermission` messages — generic defaults often get rejected.

### 🔴 REL-2 — Account deletion flow missing (App Store blocker + GDPR)
- **Verified:** No account-deletion endpoint or screen found.
- **Impact:** Apple **requires** in-app account deletion for any app with account creation. Also a GDPR "right to erasure" obligation.
- **Fix:** Add `DELETE /api/users/me` (soft-delete or hard-delete + anonymize related rows within tenancy rules) and a "Delete account" action in Settings with a confirmation.

### 🟠 REL-3 — Set Supabase Storage prod env vars
- Uploads only persist when `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` are set on Render (`server.js:390` warns and falls back to ephemeral disk otherwise). Verify these are set in production or product images silently vanish on redeploy.

### 🟠 REL-4 — Deploy topology risk (Render free tier)
- Single instance, cold starts ~30-60s (there's client retry logic, but no keep-warm). Migrations run in the Build Command. **No keep-warm pinger exists.**
- **Fix:** Add an external cron pinging `/api/health` every 10-14 min to prevent cold starts, or upgrade the Render plan. Confirm Supabase automated backups are on.

### 🟡 REL-5 — CI & test gaps
- CI runs lint + 4 backend smoke tests. **Add, in priority order:** (1) tests for the **payment webhook** (forgery rejection) once SEC-1 is fixed, (2) **auth/tenancy** tests (can user A read user B's order?), (3) **money math** (invoice ledger, partial payments), (4) a `tsc` gate once errors clear, (5) an EAS **build** gate.

### 🟡 REL-6 — Privacy/Terms are placeholders
- `PrivacyScreen.tsx` and `TermsScreen.tsx` exist but contain placeholder copy (per project notes). Apple also wants a **hosted** privacy-policy URL. Write real copy and host it before submission.

### App-store facts (verified ok)
- Bundle IDs consistent: `com.noupro.app` (iOS + Android). Version `1.0.0`, build `1`. Android permissions (CAMERA, POST_NOTIFICATIONS) declared. These are fine — just bump build numbers per submission.
