## 0. Executive summary

The app is **far more built than the docs suggest** — most feature screens (cart, orders, invoices, deliveries, events, returns, transfers, procurement, feedback) are real, DB-backed, with proper loading/error/empty states. The earlier "100% mockup" features have largely been finished.

The real risks now fall into four buckets:

1. **💰 Money/security holes** — payment webhook can be forged, paid tiers can be granted without payment, recurring billing never charges, a few endpoints leak other companies' data. **These are the only true ship-blockers.**
2. **🎭 Fake data shown to real users** — the Business Home dashboard and the Explore tab silently render *convincing fabricated data* when the backend returns nothing. This is the most likely thing to embarrass you in a demo.
3. **🔌 A handful of dead buttons / 404s** — small, concrete, quick to fix.
4. **🧹 Tech debt & half-built UI** — invoice template settings, supplier "add product", contact messages, 7 "Coming soon" sidebar items, modal migration, ~176 TypeScript errors. Not urgent, but visible.

## 1. 💰 Ship-blockers (do these before charging real money)

| # | Issue | Where | Status |
|---|-------|-------|--------|
| **B1** | **Peach payment webhook accepts forged requests.** No signature/HMAC check. Anyone who POSTs a fake "success" event with a guessed `checkoutId` can grant themselves a paid subscription or mark an invoice PAID — for free. | `backend/server.js:14155` | ✅ Confirmed still open |
| **B2** | **`PATCH /companies/:id/subscription` grants paid tiers with no payment.** A second free-upgrade path — writes `subscriptionTier` straight to the DB. | `backend/server.js:2642` | ✅ Confirmed |
| **B3** | **Recurring billing never charges the card.** `chargeStoredCard` has zero callers and there's no renewal cron — paid subscriptions silently lapse instead of renewing. | backend billing | 📄 AUDIT P0-2 — re-confirm |
| **B4** | **Endpoints leak other companies' data** (no tenant check): `GET /companies/:id/access/locations` (`:11220`, enumerates any user's access in any company), `GET /locations/:locationId` (`:3409`, returns any company's location), `GET /users/:userId/contacts` (`:10584`, missing self-check). | `backend/server.js` | ✅ Confirmed |
| **B5** | **Uploaded images vanish on every deploy** (multer → ephemeral Render disk). Code to move to Supabase Storage is written but **not live** — still needs the bucket created + service key + Render env var set. | backend uploads | 📄 P0-5 — pending manual go-live |

**Recommended fix order:** B1 → B2 → B4 (all small code changes) → B5 (manual ops) → B3 (needs a scheduler).

## 4. ⚠️ Backend reliability & remaining security

| # | Issue | Where | Notes |
|---|-------|-------|-------|
| **R1** | **No async-error safety net.** 13 `async` routes have no try/catch, and there's no `express-async-errors` shim — so a DB error makes the **request hang until timeout** instead of returning 500. | `backend/server.js` (companies/locations CRUD cluster ~`:2110`–`:3580`) | One-line fix: `require('express-async-errors')` at the top of server.js |
| **R2** | **Public storefront routes are unthrottled.** `POST /public/locations/:id/orders`, `GET /public/locations/:id`, `/products` have no rate limiter — open to anonymous order-spam. | **R3** | **`GET /api/companies` has no auth** (vs the auth'd `GET /api/businesses` which returns similar data). Likely unintended public exposure. | `backend/server.js:2697` | Confirm intent; add auth if not intentional |

## 5. 📊 Data-layer & integration TODOs

- **Invoice payments aren't truly tracked.** Recording a payment just PATCHes status (PAID / PARTIALLY_PAID + paidAmount); there's no per-payment ledger. Partial-payment history is faked. (`src/features/invoices/invoices.service.ts:107`, needs a backend endpoint.)
- **Dual push-token systems** — the `DeviceToken` table (dead, the 404 above) vs `PushToken` (live). Consolidate; add Expo push-receipt pruning so dead tokens don't accumulate.
- **Dual inventory source of truth** — `Product.stockQuantity` vs `Stock.qtyOnHand`. Pick one.
- ✅ **DONE 2026-06-27 — `orderAutomation` Task 4 + scheduler.** Implemented stuck-pending-order push alerts (`notifyStuckPendingOrders` → `pushService.sendToUsers`, notifies each business's accepted admins) and added a **daily** scheduler in `server.js` (alerts only; auto-cancel stays manual via `POST /api/automation/orders`, gated by the `autoCancel` option). No migration.
- ✅ **DONE 2026-06-27 — `/api/feed` no longer masks DB errors.** The catch now returns `500 errorResponse('Failed to fetch feed','FEED_ERROR')` instead of an empty `200`; the `useFeed` hook already surfaces the error.
- **Business hours hardcoded** to "Not configured" on the own-business profile (`BusinessProfileOwnScreen.tsx:556`) until a DB field is added.
- ✅ **DONE 2026-06-27 — `via.placeholder.com` removed.** New `ImageOrPlaceholder` shared component (themed `imagePlaceholder` bg + icon) replaces all 8 third-party fallbacks across Explore/Products search.
- **Real staff-invite emails (deferred from R4).** The invite-by-email + resend UI was removed (2026-06-27) because it never sent email (token generated then discarded; invitees never notified). To restore it: add a `sendStaffInviteEmail()` (mirror `sendPasswordResetEmail`, `backend/server.js:91`), persist + validate an invite token, send the email on both invite and resend, then re-add the invite-by-email Section in `InviteStaffScreen.tsx` and the resend action in `TeamManagementScreen.tsx` (+ `resendInvite` in `team.service.ts`). The dormant `POST /users/invite` backend route is kept for this. Until then the working invite path is **Copy link → join-request → admin approve**.
